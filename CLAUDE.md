# secretdef

Open standard and TypeScript library for declaring environment secret requirements. Like `@types` for secrets.

**Website:** https://secretdef.com
**Repo:** https://github.com/iplanwebsites/secretdef

## Monorepo structure

```
packages/secretdef/          # Core library (npm: secretdef)
packages/@secretdef/*/       # Community secret definitions (aspirational, not fully verified)
packages/create-secretdef/   # CLI tool (Phase 2, not yet implemented)
apps/web/                    # Docs + brochure site (Vike + React + Tailwind)
apps/web/data/               # Generated data files (registry.json, integrations.json)
scripts/                     # Generation & scanning scripts
docs/                        # PRD and planning docs
```

## Commands

```bash
pnpm install                        # Install all dependencies
pnpm build                          # Build all packages (turbo)
pnpm test                           # Run all tests (turbo)

# Core library only

# Web app

node scripts/generate-packages.mjs  # Generate new @secretdef/* packages
node scripts/scan-sdk-env.mjs       # Clone SDK repos, scan for process.env, update registry.yaml
```

## Data pipeline

The website reads from two data files in `apps/web/data/`:

| File | Source | Purpose |
|------|--------|---------|
| `registry.json` | Generated by `scripts/generate-registry.mjs` | All @secretdef/* packages (1300+), with featured metadata |
| `integrations.json` | Hand-edited via GitHub PRs | Libraries that ship native `defineSecrets()` calls |

### registry.json pipeline

```
registry.yaml (curated: title, npm, repo, hasEnvVars, dateEvaluated)
       +
packages/@secretdef/* (filesystem scan of all packages)
       ↓
scripts/generate-registry.mjs
       ↓
apps/web/data/registry.json  →  { name, title?, npm?, featured? }[]
```

- `featured: true` = entry exists in `registry.yaml` with `hasEnvVars: true`
- The web app's `prebuild` script runs `generate-registry.mjs` automatically before each build
- To regenerate manually: `pnpm generate-registry`

### integrations.json

Separate file for SDKs/libraries that natively ship `defineSecrets()`. Users submit PRs to `apps/web/data/integrations.json`.

### registry.yaml

Tracks curated SDK candidates with verification status. Fields: `name`, `title`, `npm`, `category`, `repo`, `hasEnvVars`, `dateEvaluated`. Used by:
- `scripts/generate-registry.mjs` — reads featured metadata
- `scripts/scan-sdk-env.mjs` — clones repos, scans for `process.env.*`, writes back `hasEnvVars` + `dateEvaluated`

## Core library API

Core exports:

```ts
import {
  defineSecrets,
  validateSecrets,
  useSecret,
} from "secretdef";
import type { SecretSpec } from "secretdef";
```

- `defineSecrets(specs)` — Returns normalized `Record<string, SecretSpec>` and always auto-registers to the global registry.
- `validateSecrets(specs?, env?, options?)` — Validates an explicit map, or the auto-registry if no map passed. Returns resolved `Record<string, string>`.
- `useSecret(key, specs?)` — Reads a single secret. Accepts explicit map or falls back to auto-registry.
- `enableAutoRegister()` — **Deprecated no-op.** Auto-registration is always on. Kept for backward compatibility.

### Two consumer styles

**Auto-register (easy, default):** Import your `secrets.ts` files, then call `validateSecrets()` with no args. `defineSecrets` always registers globally.
**Explicit spreading (structured):** Spread specs into `validateSecrets({ ...app, ...db })`. Full control over which specs are validated.

The primary use case is **app developers** defining their own secrets in `secrets.ts` files. SDK/module authors can also ship secret definitions alongside their packages.

## Core library source files

- `src/types.ts` — `SecretSpec`, `RegisteredSecret`, `ValidateOptions`
- `src/registry.ts` — Global `Map`, `register()`, `clearRegistry()` (auto-registration always on; `enableAutoRegister()` is a deprecated no-op)
- `src/caller.ts` — Stack trace parsing to extract caller file path
- `src/resolve.ts` — Resolves a spec against env + environments
- `src/validate.ts` — `validateSecrets()` with error/warn output (❌/🔑/🔧 markers, .env block, optional summary)
- `src/accessor.ts` — `useSecret()` + `SecretNotAvailableError` + `SecretValidationError`
- `src/generated.ts` — Writes `.secretdef/generated-*.json` snapshots; `findProjectRoot()` locates nearest `package.json` for monorepo-safe placement
- `src/index.ts` — Public API barrel

## @secretdef/\* packages

Each package is a single `index.ts` exporting `const secrets = defineSecrets({...})` with a `package.json` that peer-depends on `secretdef`. Uses `workspace:*` protocol locally.

To add a new package: create `packages/@secretdef/<name>/index.ts` and `package.json`.

## Key design decisions

- `defineSecrets` returns pure data — no Proxy, no magic. Always auto-registers to global registry.
- Production mode: missing required secrets → error table → `process.exit(1)`.
- Non-production: missing secrets → warning table → server starts. Throws only on access via `useSecret`.
- Error messages are structured for both humans and AI agents: include env var name, description, dashboard URL, defining file, current environment.
- Zero dependencies. Target <2KB minified.
- `@secretdef/*` packages use peer deps to ensure single global registry.

## Conventions

- Core library changes require tests. Run `pnpm --filter secretdef test` before committing.
- All `@secretdef/*` packages follow the same structure: single `index.ts` with `export const secrets = defineSecrets({...})`.
- Every secret spec should include a `description` and a `dashboard` URL pointing to the service's dashboard/settings page.

### Common pitfalls

- **"no global state"** — Wrong. `defineSecrets` always auto-registers to the global registry.
- **"Registered by" vs "Defined in:"** — User-facing label is `Defined in:`, internal field is `registeredBy`.
- **`choices?: string[]`** — Actual type is `readonly string[]`, not mutable.
- **SecretNotAvailableError format** — Must match actual `accessor.ts` output exactly.
- **devDefault** — Resolves silently, no warning output when it kicks in.
- **`enableAutoRegister()`** — Deprecated no-op. Don't document it as if it does something.
- **SKILL.md** — Has 3 copies. Edit `skills/secretdef/SKILL.md` then run `pnpm sync-skills`.
- **Terminal output emoji** — Use: ❌ (error header), ⚠️ (warn header), 🔑 (missing), 🔧 (invalid).
- **Pluralization** — "1 secret problem" not "1 secret problems" or "problem(s)".

## @secretdef/* package build

Each `@secretdef/*` package builds with `tsup` → `dist/` containing ESM, CJS, and DTS outputs:

```
dist/index.js      # ESM
dist/index.cjs     # CJS
dist/index.d.ts    # TypeScript declarations (ESM)
dist/index.d.cts   # TypeScript declarations (CJS)
```

Every package has 4 files:

| File | Purpose |
|------|---------|
| `index.ts` | Source — `export const secrets = defineSecrets({...})` |
| `package.json` | Metadata, exports map, peer dep on `secretdef` |
| `tsup.config.ts` | Build config (ESM + CJS + DTS, externals) |
| `tsconfig.json` | Scoped `include: ["index.ts"]` — prevents DTS from picking up sibling packages |
| `README.md` | Install/usage instructions + secrets table |

Build commands:

```bash
pnpm build                              # Build ALL packages via turbo (~5 min cold, cached after)
pnpm --filter @secretdef/openai build   # Build a single package
```

**Important:** The `tsconfig.json` in each package scopes TypeScript to only that package's `index.ts`. Without it, tsup's DTS generation walks the root tsconfig and fails on missing `.d.ts` files from unbuilt sibling packages.

## Mass-generating @secretdef/* packages

### Generator script

The primary tool is `scripts/generate-packages.mjs`. It contains a large `services` array and generates all 4 files per package.

```bash
node scripts/generate-packages.mjs      # Generate new packages + backfill missing files
```

The script:
- **Skips** packages that already exist on disk (safe to re-run)
- **Skips** packages in the `SKIP` set (known duplicate env var conflicts)
- **Backfills** missing `tsup.config.ts`, `tsconfig.json`, and `README.md` for existing packages

### Adding a new service

Add an entry to the `services` array in `scripts/generate-packages.mjs`:

```js
{
  name: 'my-service', title: 'My Service',
  secrets: {
    MY_API_KEY: { description: 'My Service API key', dashboard: 'https://dashboard.myservice.com/api-keys' },
    MY_OPTIONAL_VAR: { description: 'Optional config', dashboard: 'https://dashboard.myservice.com', required: false },
    MY_URL: { description: 'Service URL', dashboard: 'https://myservice.com', required: false, validate: 'url' },
  },
},
```

Rules:
- Primary API key is `required: true` (default, omit the field)
- Optional config vars get `required: false`
- `dashboard` is the URL to the service's settings/API keys page
- Use `validate: 'url'` for connection strings and URLs

Then run:

```bash
node scripts/generate-packages.mjs   # Generate the package
pnpm install                          # Update lockfile
pnpm --filter secretdef test          # Verify no duplicate env var keys
pnpm build                            # Build (turbo caches unchanged packages)
```

### Duplicate env var keys

The global registry throws on duplicate keys. If two packages declare the same env var (e.g. `AWS_ACCESS_KEY_ID` in both `aws` and `aws-bedrock`), tests will fail. Fix by:
1. Removing the duplicate key from the more specific package
2. Adding the package name to the `SKIP` set if the entire package is redundant

### Research workflow for new services

1. Identify the service's SDK or API
2. Find env vars: grep for `process.env.` in the SDK repo, check README/.env.example
3. Find the dashboard URL where users manage their API keys
4. Add to the `services` array and run the pipeline above
